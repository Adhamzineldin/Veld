What Veld Actually Does

Core Philosophy
You write one .veld contract file describing your API.
Veld generates: (1) a typed SDK the frontend imports, (2) interface stubs the backend implements.
The developer fills in business logic. Veld never touches it. Regeneration is safe.

The Pipeline

The Go core engine processes your .veld file through 5 deterministic stages.

📄
Source
.veld contract file written by developer
✂️
Lexer
Tokenizes source into keywords, identifiers, symbols
🌳
Parser
Builds AST from token stream, validates grammar
✅
Validator
Checks type refs, missing models, schema conflicts
📦
AST JSON
Language-agnostic intermediate representation
🔌
Emitters
Plugins read AST → generate code for target stack

The .veld Contract File .veld 
// auth.veld — source of truth

model LoginInput {
  email:    string
  password: string
}

model RegisterInput {
  email:    string
  password: string
  name:     string
}

model User {
  id:    string
  email: string
  name:  string
}

module Auth {

  action Login POST /auth/login {
    input      LoginInput
    output     User
    middleware RateLimit
  }

  action Register POST /auth/register {
    input  RegisterInput
    output User
  }

  action Logout POST /auth/logout {
    output SuccessResponse
    middleware AuthGuard
  }

  action Me GET /auth/me {
    output User
    middleware AuthGuard
  }

}

What Veld Generates

Veld writes these files. The developer never manually edits them.

GENERATED / 01
DTOs & Types
Input/output types for every action. Generated in target language.
LoginInput, RegisterInput structs
User, SuccessResponse types
Validation annotations baked in
Never edited by developer
GENERATED / 02
Service Interface
Abstract interface the developer implements in their own file.
interface IAuthService
One method per action
Typed input → typed output
Developer implements this
GENERATED / 03
Route Wiring
Framework-specific route binding. Calls the service interface.
Express router / Flask blueprint / Laravel routes
Middleware applied automatically
Input parsed & validated
Calls service.Login(input)
GENERATED / 04
Frontend SDK

Return type inferred
No magic strings ever

import { Router } from 'express'
import { IAuthService } from '../interfaces/IAuthService'
import { rateLimitMiddleware, authGuard } from '../middleware'

export function authRouter(service: IAuthService) {
  const router = Router()

  router.post('/auth/login', rateLimitMiddleware, async (req, res) => {
    const input = req.body as LoginInput
    const result = await service.Login(input)
    res.json(result)
  })

  router.post('/auth/register', async (req, res) => {
    const result = await service.Register(req.body)
    res.json(result)
  })

  router.get('/auth/me', authGuard, async (req, res) => {
    const result = await service.Me(req.user.id)
    res.json(result)
  })

  return router
}

def create_auth_blueprint(service: IAuthService):

    @bp.route('/auth/login', methods=['POST'])
    @rate_limit
    def login():
        data = request.get_json()
        result = service.login(data)
        return jsonify(result)

    @bp.route('/auth/register', methods=['POST'])

        data = request.get_json()
        result = service.register(data)

from app.repositories.user_repo import UserRepo
from app.utils.crypto import hash_password, verify_password

class AuthService(IAuthService):

    def login(self, data: dict) -> User:
        # YOUR LOGIC — DB, hashing, JWT, etc.
        user = UserRepo.find_by_email(data['email'])
        if not user:
            raise ValueError('User not found')

        if not verify_password(data['password'], user.hash):
            raise ValueError('Invalid credentials')

// ✅ Build on top of the generated SDK naturally

import { api } from '../generated/client/api'
import type { User, LoginInput } from '../generated/types/auth'

export function useAuth() {
  const [user, setUser] = useState<User | null>(null)
  const [loading, setLoading] = useState(false)

  const login = async (input: LoginInput) => {
    setLoading(true)
    try {
      const u = await api.Auth.Login(input)
      setUser(u)
    } finally {
      setLoading(false)
    }
  }

  return { user, loading, login }
}

Hard Rules to Never Break

Rule 1 — Regeneration Safety
Veld NEVER writes to files outside /generated/. If a dev edits a generated file, warn them. If veld would overwrite business logic, abort with an error.
Rule 2 — Core Engine Isolation
The parser/validator NEVER imports emitter code. Emitters NEVER import parser internals. Only the AST JSON passes between them. Violating this means adding Flask support breaks the TypeScript emitter.
Rule 3 — Deterministic Output
Running veld generate twice on the same .veld file must produce identical output. No timestamps, no random IDs, no environment-dependent generation. This enables git diffs and CI checks.
Rule 4 — Contracts Are Source of Truth
If the .veld file says LoginInput has {email, password}, every emitter must agree. No emitter can invent extra fields. The AST is law.
Rule 5 — Framework Freedom
Veld generates glue code that plugs INTO the framework — never replaces it. Dev must be able to use Express middleware, Flask decorators, Laravel Eloquent, etc. exactly as they normally would.
